## Section 0 — Slack-ready Summary

**TL;DR**: Both bugs share the same root cause: **WF-08 has no concept render step**. TR-001 feeds raw text directly to GN-009, so GN-009 takes the **text-to-video** path (no source image). Kling's text-to-video endpoint cannot reliably produce cinematic camera motion or interior walkthroughs without a visual reference. The exterior shot becomes a near-static establishing frame; the interior shot never enters the building because Kling has no interior image to animate from. **Fix: insert a GN-003 concept render node between TR-001 and GN-009 in WF-08 so the image-to-video path is used** (which already has Phase 3 GPT-Image-1 interior reference generation). Alternatively, add a fallback interior-image-generation step to `submitDualTextToVideo`. Estimated fix: 1-2 days for Path A, 3-5 days for Path B.

### A.6 Why the exterior is near-static — root cause analysis

Three compounding factors:

**Factor 1: Camera motion instruction is a single vague sentence.**

The text-to-video exterior prompt ends with just:
> "Cinematic exterior views including front, sides, back, and top view of the building."

Compare this to the **image-to-video** exterior prompt (`buildExteriorPrompt` at `video-service.ts:300-320`) which has 10+ lines of detailed camera trajectory:
> "Camera starts at the front elevation — slow cinematic dolly toward the building entrance...
> Camera smoothly orbits to the side elevations, revealing the building's depth...
> Continues to the back elevation...
> Camera rises on a sweeping crane shot to a dramatic top-down aerial perspective..."

For text-to-video, where Kling has to generate BOTH the visual content AND the camera path from scratch, the instructions need to be MORE detailed, not less. A single sentence gives Kling no trajectory to follow.

**Factor 2: 5-second duration is too short for text-to-video.**

Kling text-to-video needs time to: (a) establish the building visually, (b) develop camera motion. At 5s, there's barely time for an establishing frame. Image-to-video can start moving immediately because the visual content is already provided.

**Factor 3: PDF rendering brief ≠ building description.**

The first 1900 chars of the PDF text extracted by `pdf-parse` include:

```
RENDERING BRIEF — MARXSTRASSE 12, GAGGENAU
...
Three ground-floor apartments — twelve photorealistic interior renderings
...
Scope: 12 interior renderings — 4 per unit across 3 units
Style: Photorealistic, warm, inhabited — lifestyle, not staged-empty
...
Architectural baseline:
• Floor: solid oak engineered plank, medium-warm tone
• Walls: warm chalk-white...
```

This is a **photography direction document**, not a building specification. It describes how to photograph interiors, not the building's architectural form. Kling receives interior design specifications and is asked to generate an "exterior view" — a semantic mismatch.

**NONE.** Text-to-video has **no image field at all**. The request body (`video-service.ts:542-549`) has no `image` parameter.

**This is THE critical distinction from the image-to-video path.** In the image-to-video path at `gn-009.ts:519-557` (Phase 3), the handler generates a dedicated GPT-Image-1 eye-level interior reference image:

This interior reference is then passed to `submitDualWalkthrough` at `video-service.ts:769`:

```typescript

// ...

```

The text-to-video path (`submitDualTextToVideo`) **has no equivalent**. Both text-to-video tasks get identical treatment — pure text, no image reference at all.

### B.3 Does the pipeline generate an INTERIOR reference image anywhere?

**No — not for the text-to-video path.** The interior reference generation exists ONLY in the image-to-video branch (`gn-009.ts:519-557`). The text-to-video path at `gn-009.ts:687-792` calls `submitDualTextToVideo` directly with no image generation step.

The `cinematic-pipeline.ts` module has `generateLifestyleImage()` (line 567) which generates exactly this kind of interior reference via GPT-Image-1, but it is **never called from the text-to-video path**.

### B.4 Does the PDF-brief-summary forward INTERIOR room data downstream?

**Partially.** TR-001 at `tr-001.ts:75-118`:
- `parseBriefDocument` extracts programme spaces (rooms + areas) from the PDF
- Output includes `_raw.rawText` (first 12,000 chars of extracted text) and `_raw.programme` (structured room list)
- `formattedContent` includes "PROGRAMME REQUIREMENTS" with room list

GN-009 picks up `rawText` at line 49 (`originalPdfText = raw?.rawText`) and uses it as `buildingDesc`.

However, `buildExteriorTextPrompt` and `buildInteriorTextPrompt` **only use `buildingDescription` (= rawText)**. They don't use the structured `programme` array or `roomInfo`. The PDF text gets truncated to 1900 chars, which captures the rendering brief's "Project Context" and "Global Direction" sections — mostly about photography style, not room-by-room architecture.

For this specific PDF (Marxstraße 12 rendering brief), the first 1900 chars include:
- Project metadata (address, building age, energy class)
- "12 interior renderings — 4 per unit across 3 units"
- Style direction (photorealistic, Apartamento magazine, etc.)
- Material palette (oak floors, chalk-white walls)

The **individual room specifications** (WE 01bb kitchen-dining, WE 02bb living room, WE 03bb bedroom) appear LATER in the PDF — past the 1900-char cutoff. So the interior-specific data that could help Kling generate actual interior content **is truncated away**.

### B.5 Is there a hardcoded camera trajectory?

**No hardcoded camera trajectories in the text-to-video path.** The text-to-video prompts (`buildExteriorTextPrompt`, `buildInteriorTextPrompt`) have no camera trajectory constants. They rely entirely on the single-sentence instruction at the end.

By contrast, the `cinematic-pipeline.ts` (not used here) has hardcoded room descriptions ("L-shaped grey sectional sofa", "golden retriever sleeps curled up") — see Section D for the full audit.

### B.6 Why the interior never goes inside — root cause analysis

**Kling text-to-video cannot produce interior walkthroughs from text alone.** Without a visual reference showing what the interior looks like, Kling's text-to-video model:

1. Generates an exterior establishing shot (a building facade matching the description)
2. Attempts to "walk through" by zooming toward/into the facade
3. Never truly enters the building because it has no interior visual data to transition to

This is an inherent limitation of text-to-video for architectural interior content. The model would need either:
- An interior reference image (as the image-to-video Phase 3 provides via GPT-Image-1)
- A multi-frame keyframe sequence showing exterior → transition → interior
- Or Kling's image-to-video endpoint with a dedicated interior start frame

## Section E — Discrepancies

### E.1 Duration: UI shows ~16.5s but shots sum to 15s

**Hypothesis**: The user-reported "16.5s" value could come from:

1. **Kling's actual video duration** differing from the requested duration. Kling's `task_result.videos[].duration` field reports the actual rendered duration, which can slightly exceed the requested `5` or `10` seconds. The code uses the **requested** duration (5+10=15) for segment records, not Kling's actual duration.

2. **Stale metadata in the Execution.tileResults JSONB**: If the BG job's incremental patch (`video-job-service.ts:287-318`) ran mid-flight and wrote a `completedDuration` from only one segment, then the terminal patch overwrote it with 15s. A race condition could leave an intermediate value visible briefly.

3. **The stat strip** (`derive-stat-strip.ts:113`) reads `v.durationSeconds` directly — if the VideoJob row's `totalDurationSeconds` was computed differently (e.g., from Kling's reported actual durations), it could show a non-15 value.

**Verification required**: Query the `Execution` row (`cmohc8pt4000204l45ya55cy3`) and the associated `VideoJob` row. Check `totalDurationSeconds` on the VideoJob and `durationSeconds` in the tileResults video artifact. This cannot be verified from code alone.

### E.2 GPT-Image-1.5 vs GPT-Image-1

The text-to-video path uses **neither** — it generates no images at all.

The image-to-video path's Phase 3 interior reference generation uses `generateLifestyleImage` which calls `client.images.edit` with `model: "gpt-image-1"` (`cinematic-pipeline.ts:688-695`). This is GPT-Image-1 (not 1.5 — there is no "1.5" in the codebase).

If the fix involves adding interior image generation to the text-to-video path, GPT-Image-1's `images.generate` (not `images.edit`, since there's no source image to edit) should be used. The `images.edit` API in `generateLifestyleImage` requires a source image (`floorPlanRef`) which doesn't exist in the text-to-video case.
